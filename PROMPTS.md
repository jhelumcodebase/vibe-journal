This document records the exact sequence of prompts used to design, build, debug, and deploy VibeJournal. Someone else can use these prompts to recreate this exact application.

## 📅 Chronological Prompt Log

### Phase 1: Environment Setup & Project Scaffolding
1. **Initialize Project Workspace**:
   *Scaffold a new ADK 2.0 python prototype using standard CLI toolings:*
   ```bash
   agents-cli scaffold create vibe-journal --agent adk --prototype
   ```
2. **Setup Pydantic State Schemas**:
   > Define the state tracking models. Inside `app/state_schema.py`, write two distinct Pydantic models:
   > - `SkillMatrix(BaseModel)`: Include integer fields scaled 1-100 for `ai_foundations_score`, `agentic_ai_score`, and `system_design_score`. Include string fields for `primary_weak_area`, `improving_trends`, and `key_takeaway_logged`.
   > - `InterviewPrep(BaseModel)`: Include string fields for `mock_interview_question`, a string list for `hidden_grading_rubric` containing 3-4 structural criteria, and an `ideal_architectural_answer` text block field.
   >
   > Ensure these profiles are cleanly exported for use in our graph nodes.

### Phase 2: Core Agent Workflow & Firestore Integration
3. **Build the ADK Multi-Agent Graph**:
   > Modify `app/agent.py` to create an asynchronous ADK 2.0 multi-agent graph workflow containing three distinct nodes:
   > - **Node 1 (`ChatResponder`)**: Initialize a standard Agent mapping to `gemini-2.5-flash`. Instruct it to act as an encouraging AI/Software Engineering mentor providing immediate answers to incoming user prompts based on conversational history context.
   > - **Node 2 (`SkillEvaluator`)**: Initialize an analytical Agent mapping to `gemini-2.5-pro`. It must ingest the full chat log, audit the user's technical gaps, and strictly populate the structured JSON data schema using our `SkillMatrix` model.
   > - **Node 3 (`InterviewCoach`)**: Initialize a Principal Interviewer Agent mapping to `gemini-2.5-pro` that reads the evaluated weakness from Node 2 and outputs a tailored interview simulation mapping directly to the `InterviewPrep` schema structure.
4. **Persistent Firestore Cloud Integration**:
   > Update our storage topology to handle persistent cloud tracking. Inside our ADK graph lifecycle hooks, integrate `google-cloud-firestore`. Write an asynchronous database write method that captures the structured dictionary outputs from Node 2 and Node 3. Whenever a workflow turn completes, append a new historical document containing the timestamp, user query, AI reply, and `key_takeaway_logged` string directly into a sub-collection named `historical_logs` under a unique parent `session_id` document key. Ensure all database calls are securely isolated in a safe `try/except` block to keep UI interactions fast.

### Phase 3: Frontend Dashboard & Quota Handling
5. **Streamlit UI & Docker Scaffolding**:
   > Build the frontend presentation layout. Inside `app/main.py`, construct a unified, dual-panel Streamlit layout (`st.set_page_config(layout='wide')`). On the left side, build a chat input box that captures user queries, runs the active ADK workflow, and streams replies. On the right side, implement visual metric progress bars representing the user's `SkillMatrix` scores along with an expandable drawer (`st.expander`) displaying the hidden interview grading keys.
   >
   > Generate an optimized production `Dockerfile` pinning Python 3.11, the required `google-adk`, `google-cloud-firestore`, and `streamlit` libraries.
6. **Retry Policies for 429 Quota Exhaustion**:
   > Fix the 429 `RESOURCE_EXHAUSTED` error by adding robust retry logic to our models in `app/agent.py`. Update the configuration for our Gemini models (specifically the `gemini-2.5-pro` instances inside `SkillEvaluator` and `InterviewCoach`). Configure the model instances using the native ADK retry options or an `HttpRetryOptions` wrapper. Ensure that upon hitting a 429 status code, the application automatically implements an exponential backoff with jitter, retrying up to 5 times before failing.
7. **Vertex AI Global Endpoint Routing**:
   > Check our environment variables and backend initialization settings. If our project initializes Vertex AI or the Gemini API using a rigid regional endpoint, update the initialization config to leverage global model routing (e.g., using global endpoints instead of pinpointing a single region). This will distribute our request bursts across the global capacity pool and reduce regional 429 bottlenecks.

### Phase 4: Threading Mismatches & UI Refinements
8. **Filter Event Streams**:
   > The main chat panel streams all events returned by `runner.run_async()`, including the raw JSON schemas generated by the `SkillEvaluator` and `InterviewCoach` subagents. Modify the event streaming loop in `app/main.py` so that only events authored by `ChatResponder` are appended to the chat bubble, keeping the JSON structures hidden in the chat history while still caching and updating the analytics widgets.
9. **Decouple Event Loops (Streamlit Threading)**:
   > When a user sends a chat message, calling `asyncio.run(run_chat_turn(prompt))` inside Streamlit crashes with a `RuntimeError: Task got Future attached to a different loop` due to conflicts between Streamlit's script-runner thread loop and the `aiohttp` session inside the `google-genai` client.
   >
   > Refactor `main.py` so that the ADK `runner.run_async(...)` call runs in a background thread with its own fresh event loop on every chat turn, and streams events back to the Streamlit main thread via a `queue.Queue` so we can update `st.empty()` in real time.
10. **Lazy Agent Factory (Global Loop Lock Mismatch)**:
    > We are still hitting a loop mismatch error because the LlmAgents and Gemini models are instantiated once at module-import time, locking their clients to the import event loop.
    >
    > Refactor the agent definitions in `app/agent.py` to use a lazy factory function `get_workflow() -> Workflow`. In `app/main.py`, call this factory inside `run_workflow_in_thread` to dynamically instantiate the workflow and `Runner` on the background thread's active loop.

### Phase 5: Production Deployment & Verification
11. **FastAPI & Cloud Run Scaffolding**:

12. **Container PYTHONPATH Resolution**:
    > The Cloud Run container crashes on startup with `ModuleNotFoundError: No module named 'app'`. Update the `Dockerfile` to configure `ENV PYTHONPATH="/workspace"` before starting the Streamlit application.
13. **Restore Project Default Compute Service Account**:
    > Cloud Run deployment fails with `PERMISSION_DENIED: Permission 'iam.serviceAccounts.get' denied on resource (or it may not exist)`. Restore the deleted Compute Engine default service account:
    > ```bash
    > gcloud iam service-accounts undelete [RESTORED_UNIQUE_ID]
    > ```
    > Grant `roles/iam.serviceAccountUser` and `roles/iam.serviceAccountViewer` to the Cloud Build Service Account and Cloud Run Service Agent.
14. **Build & Deploy Container**:
    > ```bash

    >     --platform managed \
    >     --region us-central1 \
